如何实现一个网格世界？
### 一阶方案：直接写
``` C#
for (var i = 0; i < 10; i++) {
    for(var j = 0; j < 10; j++) {
        //在坐标(i, j)创建一个GameObject，并且设置图片
    }
}
```
简单粗暴，但不方便存档，不方便后续新功能的添加。
### 二阶方案：地块类
``` C#
interface I地块 {
    int X { get; set; }
    int Y { get; set; }
    void Draw();//在对应位置创建GameObject，并且设置图片
}
class 地块类 : I地块 {
    //实现
}
I地块[,] 所有地块 = new I地块[10, 10];
for (var i = 0; i < 10; i++) {
    for(var j = 0; j < 10; j++) {
        所有地块[i, j] = new 地块类();
    }
}
所有地块.ForEach(t => t.Draw());
```
复杂了一些，但是这样扩展性加强了很多，可以方便的存档、方便的定义诸多针对【所有地块】的行为。
而且还有一个优点：在世界大量演变的时候节约性能。这本质是：逻辑与渲染分离。逻辑可以大量更新，而渲染只需要少量更新。（一阶方案中，逻辑与渲染混合，逻辑每一次更新都需要更新渲染）渲染的性能消耗万倍于逻辑。
### 三阶方案：Gird
二阶方案中使用二维数组。C#官方提供的二维数组很初级，我们最好自己定义一个高级的。
``` C#
interface IGrid<T> : IEnumerable<T> where T : IGridNode {
    int 宽度 { get; }
    int 高度 { get; }
    T this[int X, int Y] { get; }
    void 填充(Func<T> X);
    void 随机执行(Action<T> X, int 次数, int 平均距离 = -1);//-1表示最大、即稀疏、用于生成野怪。1用于生成矿区
    void 随机曲线(Action<T> X, float 宽度, float 曲度=0);//曲度为0表示直线
    void ForEach(Action<T> X);
    List<TResult> GetRound<TResult>(int X, int Y, Func<T, TResult> Z, int 半径 = 1);//以XY为中心，将周围半径内的格子转化后加入列表
    int SumRound(int X, int Y, Func<T, int> Z, int 半径 = 1);
    int ChangeRound(int X, int Y, Func<T, bool> 筛选, Action<T> 操作, int 半径 = 1);
}
interface IGridNode {
    int X { get; }
    int Y { get; }
}
class Grid //实现
class 地块类 : I地块, IGridNode //实现
Grid<地块类> 所有地块 = new (100, 100);
```
所有网格抽象层的操作都可以在库里定义，而无需在项目里定义。
这样可以进一步正文压缩代码行数，让正文的逻辑更简洁。
基于这个架构，可以让生成树木、生成河流、生成矿区、生成地貌，都只需要一行代码，例如：
``` C#
所有地块.填充(() => new 地块类 { 土质层 = new 土地类() };
所有地块.随机执行(t => t.土质层 = new 沙漠类(), 100, 1);//创建一个占地100左右的沙漠区域
所有地块.随机执行(t => t.土质层 = new 熔岩类(), 20, 1);
所有地块.随机执行(t => t.土质层 = new 雪地类(), 100, 1);
所有地块.随机曲线(t => t.土质层 = new 淡水类(), 2.5f, 0.1f);
所有地块.随机执行(t => { if(t.土质层 is not 熔岩类) t.建筑层 = new 树木类(); }, 100);//随机创建一百个树木
for(int i = 0; i < 10; i++) {
    所有地块.随机执行(t => t.矿石层 = new 铁矿类(), 16, 1);//在十个位置生成铁矿群落，每个群落16个铁矿
}
```
这就是优雅的代码。写起来、读起来，都赏心悦目；并且Debug与扩展新功能都非常方便。
更进一步的，还可以在项目中写一个【地图类】继承自Grid<地块类>。那样的话更好看，可以消除掉上面这些参数。但此处我觉得没什么必要（不能压行、只能行内优化），就没写。
### 四阶方案：分区渲染
如果地图尺寸1000×1000，乃至更大，那么如何处理？
内存方面没什么压力。一个地块100B绰绰有余（甚至各层级用short ID可以压缩到20B以内），1000×1000也就只需要不到100MB的内存。但问题是渲染。百万地块实时演变，i9 12900+RTX4080都能卡成一秒一帧。
一般2D游戏是可以支持百万地块直接渲染的。但风神界和一般2D游戏有一个最大的区别在于，风神界是全地块实时演变的，类似于康威生命游戏。一般游戏中大多数地块都是静态的、只需要启动游戏时加载，所以它们启动慢、但启动后运行流畅。而风神界是相当于时时刻刻都在启动。
对此，一个简单的解决方案是：小位面，多位面。即，一个位面只允许100×100的尺寸。但有很多个位面，可以跨位面传送。
如此，只需要渲染玩家所在的位面即可。
但这样不太好玩，100×100太小了。

那怎么办呢？使用显卡加速？毕竟之前虽然卡顿，但基本没用到显卡，都是CPU在负重前行。
对此：不。显卡是后续留给AI用的，不要浪费在渲染上。性能关键在于两个大头：渲染、AI。我们需要先把渲染成本压缩到最低。

最终解决方案是：分区渲染。对于一个1000×1000乃至更大的位面，每100×100视为一个区域。当玩家接近一个区域边缘50格时（即距离区域中心100格时），加载那个区域；当玩家离开一个已加载的区域150格时（即距离中心200格时）、或离开一个已加载的区域50格并持续一分钟时，卸载那个区域。
